# 内容与安全政策（Content Policy）

更新日期：2026-06-24
目标产品：Ourdream.ai 克隆（18+ AI 角色扮演 / AI 伴侣平台）

## 0. 这份文档是什么

`架构 07-security-and-compliance.md` 已经定义了安全**机制**（五层审核流水线、policyCode 表、age gate / age verification 守卫、举报申诉表）。
本文件补齐这套机制目前缺失的东西：**规则本身（the WHAT）** —— 什么内容禁止、什么受限、命中阈值如何判定、用户侧申诉怎么走、年龄如何强制、分辖区验证何时触发。

定位是**产品决策 SSoT**，面向产品/运营/审核团队，也是 Safety Center 用户可读页（`/safety/*` 镜像，07 §7）的内容母本。

- 规则正文（用户可读）→ 版本化镜像进 `policy_versions`（07 §7）。
- 机器码（policyCode）/ 表结构 / 守卫 → 见 07 §4、§5、§2。
- 与 07 冲突时：本文件定义**规则与阈值**，07 定义**实现与机器码**，两者引用而非复制。

> ⚠️ 法律免责：本文件是产品政策框架，不构成法律意见。年龄验证 / 分辖区义务因司法辖区而异，**上线前必须经法务确认**（与 07 同口径）。
>
> ℹ️ 范围说明：涉未成年素材的**自动检测管线与法定上报**（哈希匹配、机构上报等）由**合规/法务侧独立负责，不在本产品/工程设计范围**。本文件只声明"未成年内容禁止"这一产品政策底线。

## 1. 禁止内容清单（硬策略，不可关闭）

**命中即拦截 + 留证（`moderation_events` 不可变）。无任何"创作自由""明显虚构"豁免，公开与私聊一律适用。**
policyCode 与 07 §4 完全一致。

| 类别 | policyCode | 一句话用户可读定义 |
| --- | --- | --- |
| 未成年 | `UNDERAGE` | 任何描绘、暗示或编码为未满 18 岁个体的性 / 暗示 / 浪漫内容；含校园设定、"快满 18""很嫩"、loli 画风、童体比例（无论标注年龄）。命中即拦截。 |
| 真人深度伪造 | `DEEPFAKE` | 用真实人物面孔 / 影像合成的内容，无论是否点名。 |
| 真实人物肖像 | `REAL_PERSON_LIKENESS` | 是、描绘或可被合理观者认出的真实个体（名人、公众人物、私人；在世或离世不足 100 年）；触发条件三选一：可识别样貌、姓名、指向特定个体的生平细节。 |
| 受版权 IP | `IP_INFRINGEMENT` | 来自受版权作品的具名角色 / 世界观 / 剧情线（如"你是 Naruto"）；风格 / 流派 / 通用原型不算。 |
| 非自愿框架 | `NON_CONSENT` | 把用户设定为非自愿施害方，或将"行为本身"呈现为卖点的场景（奴役、绑架、下药、洗脑、买卖人口、"它别无选择"）。 |
| 违法内容 | `ILLEGAL` | 在明显虚构框架之外描绘或教唆真实世界犯罪（武器制造、毒品合成、金融诈骗）。 |
| 规避尝试 | `EVASION_ATTEMPT` | 用非英语 / 编码 / base64 / 变体命名 / 恶意重复重交，把上述任一类别藏起来；判定为"底层违规 + 一条独立 AUP 违规"。 |

补充硬类别（同样不可关闭，policyCode 见 07 §4 扩展位 `PROHIBITED_OTHER` / `SELF_HARM` / `VIOLENCE_EXTREME`）：

- **乱伦 / 家庭关系**：性 / 浪漫语境下暗示血缘关系（"女儿""双胞胎"；同款多角色默认血缘除非显式声明）。
- **兽交**：与非拟人化动物的性内容（拟人角色在 AUP 内允许）。
- **粪 / 呕 / 食人（vore）**：含委婉语（"巧克力"）；vore 作为性癖无论框架一律禁止。
- **极端血腥暴力**：图 / 视频中无叙事必要的肢解、酷刑、严重伤害（`VIOLENCE_EXTREME`）。文字叙事中服务剧情的风格化暴力允许。
- **恐怖主义 / 暴力极端主义**：宣扬、教唆、招募、美化。
- **自杀 / 严重自残推广**：鼓励或教唆（`SELF_HARM`）；角色"曾经历过"的虚构承认不算。
- **进食障碍推广**："pro-ana""pro-mia"等。
- **仇恨内容**：把仇恨当人格 / 指令；基于贬低性刻板印象的种族化内容（探讨身份认同 ≠ 种族主义）。
- **宗教人物**：涉及 Jesus / Allah / Mohammed 的性 / 浪漫内容（泛宗教设定、神职原型允许）。
- **极端政治意识形态**：主要目的是鼓吹特定政治立场的角色。

> **非自愿的边界（产品关键）**：用户**从被动方**进入（被俘、被追、被支配）是**允许**的——用户对聊天保有元同意（meta-consent），可随时离开 / 改向 / 结束。禁止的是把用户设为**施害方**或把行为本身当卖点。救援弧与救援后的自愿关系允许。

## 2. 受限内容（软策略，阈值可配置）

非禁止、但需要门控或人审的灰区。阈值是**运营配置项**（建议落在 admin 控制面，ADR / 09 配置域），不写死代码。

| 受限项 | 软策略行为 | 触发人审的条件 |
| --- | --- | --- |
| 暗示性 / 边缘年龄措辞（"娇小""petite"+ 体型差） | 输入侧降权 + 提示改写，不直接硬拦 | 与未成年线索叠加时升级为 `UNDERAGE` 硬拦 |
| 公开提交的露骨度 | 公开 bar 比私聊严格（公开内容触达所有用户） | 公开角色头像 / 简介命中边缘分类器 → 入公审队列 |
| 模糊的 IP / 戏仿 | 默认允许，存疑默认下架 | 戏仿是否"明显转化"、是否公众人物 → 人审逐例判 |
| 模糊的权力不对等 / dubcon | 用户在被动方时允许 | 框架是否把非自愿当卖点存疑 → 人审 |
| 行为信号（高拒率、批量、结构相似 prompt） | 本身**不**拦内容，只抬人审优先级 | 异常高拒率账户、与历史被拦 prompt 结构雷同 |

**最低字段门槛（公开提交，对齐 why-rejected #6）**：公开简介 ≥ 200 字符、场景 ≥ 1,000 字符；不足直接 `rejected`（可补后重交，非违规）。

## 3. 审核判定标准

对齐 07 §3 五层（`input` / `output` / `metadata_behavior` / `human_review` / `community_report`）。判定原则：

| 判定 | 何时发生 | 处置 |
| --- | --- | --- |
| **input 拦截** | 用户输入（聊天 / 生成 prompt / 角色草稿）进模型**之前**，命中最强信号（未成年、明确禁类） | 模型根本不跑，直接 `CONTENT_BLOCKED` + policyCode |
| **output 拦截** | 模型产物（文本 / 图 / 视频）释放**之前**，独立于 prompt 复检 | 未释放不展示；可触发账户级复查 |
| **自动拦截（auto-block）** | 高置信命中硬类别（§1） | 立即拦截 + 留证 |
| **转人审（manual review）** | 中置信 / 灰区（§2）/ 行为信号抬优先级 / 用户举报 / 申诉 | 入 admin 队列（05），人审给 decision + policyCode |
| **社区举报触发复审** | 用户举报任意可见内容 | 按**当前**政策复审（可能比生成时更严）；未成年类**最优先** |

判定要点：

- **置信度分档**：高置信 → 自动拦截；中置信 → 转人审；低置信 → 放行但记 `moderation_events` 备查。具体阈值是配置项，按类别可调（未成年类阈值最敏感）。
- **每次判定都写 `moderation_events`**（target / layer / status / policyCode / confidence），不可变留证（07 §3）。
- **分类器看全部，人只看被标记的**：自动分类器在线处理所有内容（含私聊私生成），无人阅读；人审只看分类器标记 / 用户举报 / 信任团队升级的内容（对齐 ourdream how-it-works）。
- **审核独立于增长 / 营收**：信任团队拥有政策，产品拥有产品；判定不看 KPI。
- **不黑箱**：申诉由**人**复核（非算法、非原决策人）。

## 4. 用户侧申诉流程（产品流，非仅 admin 队列）

这是用户**在产品里看到拒绝并申诉**的完整路径，不是后台审核队列本身（队列见 05 §15）。

### 4.1 用户怎么看到拒绝原因

- 角色提交被拒 → 创建页直接显示**拒绝原因**（映射到 §1/§2 的人类可读文案）+ 对应 policyCode + 一句"怎么改"（对齐 why-rejected 的 fix-per-reason）。
- 已发布内容被移除 → 站内通知 + 原因。
- 账户被处置（警告 / 暂停 / 封禁）→ 通知 + 原因 + 申诉入口。
- 原则（07 commitments）：**我们告诉用户发生了什么、为什么** —— 不让用户猜。

### 4.2 申诉入口、SLA、结果

| 场景 | 入口 | 复核方 | SLA |
| --- | --- | --- | --- |
| 角色被拒 | 站内"申诉"动作 → `appeals` 表；正式书面走 trust 邮箱 | **不同于**原决策人的审核员 | 内容类申诉一般 **10 个工作日**内 |
| 已发布内容被移除 | 同上 + 附原审批时间 | 同上 | 同上 |
| 账户暂停 / 封禁 | 邮件 trust 渠道（附账号 + 通知 + 具体异议） | trust 团队（非一线审核） | 复杂案件更久，会书面告知 |

申诉结果三选一（对齐 ourdream appeals）：

1. **维持原判** —— 给出更清晰的政策依据。
2. **推翻原判** —— 内容恢复 / 账户恢复。
3. **修改结果** —— 例如按指定修改后可重新发布。

### 4.3 不可申诉 / 非申诉路径

- **涉未成年内容（`UNDERAGE`）永不可申诉、永不恢复、永不重交**；账户处置不可逆。涉嫌违法素材的取证/上报由合规/法务侧独立处理（不在本文件范围）。
- 对**已成文政策本身**的不满 → 走 Contact 反馈，不是申诉。
- **原样重交**（未针对拒绝原因修改）不是申诉；改了再正常重交即可。恶意近似重复重交按 `EVASION_ATTEMPT` 处理。

## 5. 角色年龄强制（硬规则）

- **角色 `age >= 18` 硬编码强制**（创建 submit 校验 CR-09，07 §3.3）。无任何例外、无"明显虚构"豁免——*Adults, only*。
- **"Teen"等分类语义 = 18+ young adult**（年轻成年人），**绝不**指未成年或未成年外观。任何童体比例、年龄退行 / "ageing"机制、loli 画风一律 `UNDERAGE`。
- 即便 MVP 用 mock 检测 provider，**年龄字段 + 关键词硬规则也必须生效**（07 §3.2，不可因 MVP 简化跳过）。
- 用户侧对应：平台双向 18+ —— 角色必为成人（本节），用户必为 18+（§6）。

## 6. 分辖区年龄验证（age gate vs identity verification）

两层分开存储、分开判定（07 §2.2 / §2.3，ADR-7）：

| 层 | 是什么 | 何时触发 | 守卫 / 存储 |
| --- | --- | --- | --- |
| **Age gate（18+ 确认）** | 首访成人内容前确认"我已满 18"（接受即可，不验身份） | **所有**用户、首次访问成人内容前（AG-01） | `requireAgeGate` / `age_gate_acceptances` + cookie |
| **Identity verification（身份年龄验证）** | 第三方核验真实年龄（face/ID/信用卡），返回二值"已验 18+"信号 | **仅受限辖区** + 风险触发（AG-05） | `requireAgeVerified` / `age_verifications`（与 age gate 分表） |

**身份验证触发辖区（对齐 ADR-7 / ourdream age-verification）**：

- **英国（UK Online Safety Act）**：highly-effective age assurance 要求 → 强制身份验证。
- **澳大利亚（eSafety Commissioner）**：成人内容 age-assurance 期望 → 强制。
- **美国部分州 / EU 部分**：按当地法律触发（具体州 / 成员国清单需法务确认）。
- 其余辖区：默认仅 age gate；本地法要求即接入。

实现要点：

- provider 抽象（ADR-7，安全文档点名 **Go.cam**）：动态选 face/ID/credit-card 方法；平台**只收二值信号**，绝不接收底层 ID / 自拍 / 活体视频。
- **实现状态**：身份验证 MVP **暂缓**，保留表 + 守卫（默认 `not_required` 直通），不接 provider（与 07 §2.3 同口径）。
- 验证失败 → 不授权门控功能、可重试；重复失败 / 明显篡改 → 标记人审。验证后若有证据表明谎报年龄 → 暂停待查、查实即封。
- 一次验证长期有效；trust 团队可按风险（举报、跨年龄段共享账号）单独复验。
